# 架构变更检查清单

> **版本**: v1.0  
> **最后更新**: 2026-03-16  
> **适用范围**: 全仓库  
> **关联主文档**: `docs/project-architecture-foundation.md`

---

## 1. 使用方式

当改动满足以下任一条件时，提交前必须过一遍本清单：

- 新增或调整 IPC 通道
- 新增或调整任务启动 / 停止 / 同步逻辑
- 新增或调整认证、token、套餐、能力判断
- 新增独立子系统，或改变现有模块依赖方向
- 改动 shared 契约、preload 桥接、global 类型
- 改动小号互动、自动回复、数据监控、评论监听等已固化边界

目标不是“全打勾”，而是确认本次改动没有破坏已固化的真相源和依赖关系。

---

## 2. 通用检查

### 2.1 真相源

- 这次改动是否新增了第二份业务规则表？
- 是否复用了已有真相源，而不是再复制一份常量或映射？
- 如果规则需要跨语言共享，是否优先落到了 `shared/*.data.json`？

### 2.2 边界

- 是否出现 renderer 直接承担主进程权威职责？
- 是否出现页面组件直接改多套 store / runtime 状态？
- 是否出现独立子系统重新耦合进主账号体系？

### 2.3 契约

- `shared/ipcChannels.ts`、`shared/electron-api.d.ts`、preload 暴露、renderer 类型声明是否一致？
- 如果新增或改名了 IPC 通道，`electron/preload/ipcWhitelist.gen.ts` 是否已重新生成？
- 是否删除或替换了旧兼容字段？如果有，调用面是否已全部迁移？
- 是否新增了手写字符串 channel / action name，而不是复用集中定义？

### 2.4 可回归性

- 是否补了能锁住本次边界的最小测试？
- 是否更新了相关规范文档或 README 索引？
- 如果改的是高频链路，是否至少跑了 `npx tsc --noEmit` 和相关测试？

---

## 3. 分域检查

### 3.1 任务系统

- 新任务是否优先接入 `TaskManager`？
- 批量停止是否仍然统一走 `TaskStateManager` / `stopAllLiveTasks`？
- 是否避免页面层再维护一套独立任务总停逻辑？
- 是否避免任务调度器反向读取 UI store 作为唯一判断来源？

### 3.2 认证与 token

- token 是否仍只有主进程安全存储一个写入口？
- renderer 是否避免重新引入 `setTokens/getTokens` 这类旁路？
- 刷新 token 是否通过正式 auth IPC，而不是本地拼写回逻辑？
- `auth.refreshSession`、preload 暴露、renderer 调用链是否保持一致？

### 3.3 权限 / 套餐 / 能力

- 套餐静态规则是否先改 `shared/planRules.data.json`？
- 功能权限静态规则是否先改 `shared/authFeatureRules.data.json`？
- 服务端能力 DTO 和前端消费结构是否保持一致？
- 前端是否优先消费 `capabilities.feature_access`，而不是重算核心权限规则？

### 3.4 小号互动

- 是否保持独立于主账号体系？
- 是否避免把小号任务状态重新挂回全局账号任务面板？
- 与主账号体系的交集是否仍仅限可选的直播间地址获取？

### 3.5 自动回复 / 数据监控 / 评论监听

- 是否仍保持自动回复和数据监控只是 `comment-listener runtime` 的消费者？
- 是否避免一个功能启动时自动点亮另一个功能状态？
- 是否只在无消费者时才真正关闭底层评论监听？
- IPC 和文档命名是否继续使用 `commentListener` 语义，而不是回退成 `autoReply` 私有语义？
- 监听停止事件的动态前缀是否已加入 preload whitelist 生成规则？

---

## 4. 禁止事项

以下变更默认视为架构违规，除非先更新主规范并明确说明理由：

- 在 renderer 新增 token 直接写入口
- 在页面组件里新增一套独立任务总停 / 总启逻辑
- 在前端复制一份新的套餐等级或功能权限规则
- 让小号互动重新依赖主账号任务运行态
- 让自动回复和数据监控重新互相依赖
- 直接新增裸字符串 IPC 通道，而不更新 `shared/ipcChannels.ts`

---

## 5. 提交前最小动作

至少完成以下动作：

1. 自查本清单相关项。
2. 运行 `npx tsc --noEmit`。
3. 运行受影响模块的最小测试。
4. 如果变更了固化边界，同步更新：
   - `docs/project-architecture-foundation.md`
   - `README.md` 中的文档索引

---

## 6. 备注

本清单是提交前约束，不替代设计评审。  
如果本次改动需要突破当前边界，应先修改主架构文档，再落代码。
