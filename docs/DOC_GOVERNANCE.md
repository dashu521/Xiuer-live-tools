## 2. 命名规则

### 2.1 推荐命名

| 类型 | 命名格式 | 示例 |
|------|---------|------|
| 规范文档 | `*-spec.md` 或 `SPECIFICATION.md` | live-control-lifecycle-spec.md |
| 架构文档 | `*-architecture.md` | access-control-architecture.md |
| 任务清单 | `*_TASKS.md` 或 `*-checklist.md` | ONLINE_UPDATE_TASKS.md |
| 审计报告 | `*_AUDIT_REPORT.md` | SECURITY_AUDIT_REPORT.md |
| 修复报告 | `*_FIX_REPORT.md` | ACCOUNT_FIX_REPORT.md |
| 检查清单 | `*_CHECKLIST.md` | REGRESSION_CHECKLIST.md |

### 2.2 禁止命名

- `*_副本.md`、`*_copy.md` - 禁止副本文件
- `*_patch.md`、`*_complete.md` - 禁止后缀表示状态
- `*_temp.md`、`*_tmp.md` - 禁止临时文件
- `dev-*.md`（非 scripts 目录）- 禁止开发临时文档

## 3. 状态头规范

每个文档**必须**在标题后包含状态头：

```markdown
# 文档标题

> **版本**: v1.0
> **最后更新**: YYYY-MM-DD
> **状态**: 草稿 / 评审中 / 已固化 / 已完成 / 已过期 / 仅历史参考
> **负责人**: TEAM 或姓名
> **当前适用性**: 当前有效 / 部分有效 / 仅供历史参考
> **关联主文档**: 路径（如果有）

---
```

### 3.1 状态说明

| 状态 | 含义 |
|------|------|
| 草稿 | 正在编写，不可作为依据 |
| 评审中 | 等待评审，暂不可作为依据 |
| 已固化 | 评审通过，作为当前有效依据 |
| 已完成 | 任务已完成，作为历史记录 |
| 已过期 | 内容已过时，仅供参考 |
| 仅历史参考 | 归档文档，不作为实现依据 |

### 3.2 审计报告额外字段

审计类文档**必须**额外包含：

```markdown
> **问题状态**: 待修复 / 部分已修复 / 已修复待验证 / 已完成归档
```

## 4. Archive 使用规则

### 4.1 归档条件

以下文档应归档到 `docs/archive/`：

1. 问题已修复的审计报告
2. 已完成的实现报告
3. 已过期的诊断报告
4. 历史版本的修复记录

### 4.2 归档目录结构

```
docs/archive/
├── package-diagnosis/     # 打包问题诊断历史
├── regression-fix/        # 回归修复历史
├── healthchecks/          # 健康检查历史
└── implementation-reports/ # 实现报告历史
```

### 4.3 归档处理步骤

1. **保留原文档内容**
2. **添加状态头**：`状态: 已完成归档`、`当前适用性: 仅供历史参考`
3. **删除原始文件**（如果不在 archive 目录）
4. **更新引用**：确保 README 不再指向归档文档

---

## 5. 文档有效性判断

### 5.1 可作为当前实现依据

- 状态为 `已固化` 或 `当前有效` 的规范文档
- 状态为 `进行中` 的任务清单
- README 中的文档索引

### 5.2 不可作为当前实现依据

- `docs/archive/` 中的所有文档
- 状态为 `已过期` 或 `仅历史参考` 的文档
- 审计报告（仅记录问题，不定规范）
- 修复报告（仅记录解决过程，不定行为）

### 5.3 审计报告使用原则

审计报告**不是最终规范**，使用时应：

1. 先确认问题状态是否已修复
2. 对照当前代码验证问题是否仍存在
3. 最终行为以主规范和当前代码为准

## 6. 文档生命周期

```
草稿 → 评审中 → 已固化 → 已过期 → 归档
                  ↓
               已完成 → 归档
```
