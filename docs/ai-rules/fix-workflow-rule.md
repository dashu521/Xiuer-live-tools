# 四阶段问题修复流程

## 规则名称

四阶段问题修复流程

## 目的

当用户描述软件问题时，必须按固定流程处理，避免 AI 直接修改代码导致问题扩大。

## 流程概览

1. **审计 Agent**
2. **架构 Agent**
3. **修复 Agent**
4. **回归 Agent**

**默认只自动执行前两阶段，修改代码前必须等待用户确认。**

---

## 阶段1：审计 Agent

### 目标

分析问题根因，不修改代码。

### 输出

1. **问题现象总结**
2. **可能根因**（按概率排序）
3. **触发路径或复现链路**
4. **涉及代码文件**
5. **关键逻辑或函数**
6. **是否存在**：
   - 重复逻辑
   - 状态冲突
   - DEV / 打包差异
   - 旧代码残留
7. **最小修复方向**（仅建议）

### 规则

- 禁止修改代码
- 禁止给 patch
- 只做分析

---

## 阶段2：架构 Agent

### 目标

制定最小修复方案。

### 输出

1. **正确行为**
2. **当前错误行为**
3. **根因总结**
4. **最小修复策略**
5. **需要修改的文件**
6. **禁止修改的模块**
7. **风险评估**
8. **修复后的验证重点**

### 阶段结束必须输出

```
【修复计划确认】
```

并提示：

```
如需执行修改，请回复：
执行修复
```

---

## 阶段3：修复 Agent

### 触发条件

只有当用户回复：

```
执行修复
```

才允许进入此阶段。

### 执行规则

1. 只修改必要文件
2. 不修改无关模块
3. 不重构系统
4. 不新增无关功能
5. 修改范围最小

### 输出

1. **修改文件列表**
2. **每个文件修改目的**
3. **关键变更说明**
4. **潜在副作用**
5. **建议验证重点**

---

## 阶段4：回归 Agent

### 目标

验证修复效果。

### 输出

1. **基础功能验证步骤**
2. **边界场景测试**
3. **DEV 环境验证**
4. **打包环境验证**
5. **潜在回归风险**
6. **是否建议进入下一轮修复**

### 最后输出

```
【回归结果总结】
```

并询问：

```
是否进入下一轮修复？
```

---

## 全局规则

1. **最小改动原则**
2. **禁止顺手重构**
3. **禁止修改无关代码**
4. **每个阶段必须明确标记**
5. **输出必须结构化**
6. **未经用户确认禁止进入修复阶段**
7. **未经用户确认禁止自动进入下一轮**

---

## 桌面应用（Electron / Windows / macOS）问题需优先检查

- 主进程启动
- BrowserWindow 创建
- preload 路径
- 资源路径
- asar / extraResources
- 单实例锁
- 托盘逻辑
- 启动异常捕获
- DEV 与打包环境差异

---

## 使用示例

### 引用规则

```
引用规则：四阶段问题修复流程
问题：
【问题描述】
```

### 流程执行

1. AI 自动执行阶段1（审计）和阶段2（架构）
2. 等待用户确认后进入阶段3（修复）
3. 修复完成后执行阶段4（回归）
4. 根据回归结果决定是否进入下一轮
